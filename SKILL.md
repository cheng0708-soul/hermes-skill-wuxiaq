# 武侠Q传 Unity 项目

## 数据架构设计定稿（2026-06-08）

### 数据驱动铁律

```
改角色 → 写JSON      ← C#代码不动
改敌人 → 写JSON      ← C#代码不动
改关卡 → 写JSON      ← C#代码不动
改平衡 → 改config    ← C#代码不动
改战斗逻辑 → 改C#    ← 频率最低
```

### 新增武将的全流程（3步）

```
① 写 wj_XX.json（stats/growth/skills/wugong/art/spum_template/bonds）
② 跑 tools/import_wugong.py（挂技能槽，幂等可重跑）
③ 重启 Unity → GameDatabase 自动扫描加载
```

**不包含**：角色捏脸（SPUM CLI 美术线）和特效（VFX 特效线）。

### 特效跟武功走，不跟人走

```
降龙十八掌（武功数据）
  ├─ 萧峰用 → 金色龙形VFX（查武功特效表）
  └─ 郭靖用 → 金色龙形VFX（查同一个武功特效表）

设计原则：VFX/SFX/Buff 引用ID存武功JSON或特效表，
          战斗引擎按武功名查表播放，不写在角色JSON里。
```

### 三表迁移踩坑记录（2026-06-08 实战发现）

**坑 1：JSON 中旧格式与新格式共存导致执行路径对冲**

迁移期间，`wj_XX.json` 同时包含 `skills.ultimate`（旧内联对象）、`wugong[]`（旧内联对象数组）和 `talent_wugong`（新字符串引用）。CharacterImporter 的执行顺序导致 bug：

```
L44:  def.ultimateSkill = 旧内联版（从 skills.ultimate 解析）
...
L119: talentName = json.talent_wugong
L124: baseDef = WugongLibrary.Get(talentName)
```

如果 `WugongLibrary.Get()` 返回 **null**（名字不匹配），L124 只打一条 Warning，`def.ultimateSkill` **保持旧内联版不变**，`def.talentWugong` **保持 null**。战斗逻辑若读 `talentWugong` 路径则收到 null → 不触发。

**修复**：CharacterImporter 应检测到旧格式存在时不再走新格式路径，或确保新格式运行在旧格式之后且覆盖之。**迁移完成后应彻底从 JSON 移除旧 `skills.ultimate`。** 至于 wugong[] 内联对象，三表迁移后设计文档仍在使用（见前述 wugong[] 标注），是否移除需与代码实际实现对齐确认。

**坑 2：WugongLibrary.Get() 名字精确匹配导致静默失败**

JSON 里 `talent_wugong: "降龙十八掌"`，但 wugong_lib.json 的 name 字段是 `"降龙十八掌·亢龙有悔"` 或类似带后缀版本 → 精确字符串匹配返回 null → 天赋武功丢失，无报错只有一条 Warning。

**排查方法**：`Debug.Log` 打印 `[CharacterImporter] {角色名} → talent_wugong={名称}，查表结果={null/非null}`。

**坑 3：baseDef 对象直接 mutate 导致跨角色数据污染**

```csharp
// ⛔ 错误：直接篡改库中共享实例
baseDef.triggerChance = t.triggerRate;
baseDef.atkWeight = t.atkWeight;
```

WugongLibrary.Get() 返回**同一缓存对象**，萧峰的天赋覆盖会污染库中降龙十八掌的全局数据，后续郭靖读到的是萧峰版。

**修复**：`WugongLibrary.Get()` 应返回 `baseDef.Clone()`，或 CharacterImporter 在 mutate 前做深拷贝。

**坑 4：migration 分步推进但未见新格式字符就位**

方案要求 CharacterImporter 做双格式兼容（字符串→查库，对象→旧路径），但 JSON 文件未同时完成迁移。实际运行中，旧内联 `skills.ultimate` + 旧内联 `wugong[]` 仍然存在，新格式的 `talent_wugong` 和 `wugong_refs[]` 只有个别字段。CharacterImporter 的双格式兼容 ≠ 单文件内格式混用不冲突。**每条 JSON 应该全量迁移为纯新格式，不留旧字段。**

| 系统 | 来源 | 数据量 | 特点 |
|------|------|--------|------|
| **天赋武功总表** | 成哥手工补全 | 85个角色 | 角色绑定的专属武功，面板施展+spd补正→实战 |
| **武功总表** | 成哥Excel原版数据 | 109条 | 打宝掉落的通用武功，固定触发率 |

**不合并理由**：六脉神剑的天赋版（段誉专属22%+spd补正）≠ 通用版（打宝掉落42%固定）。

## spum_template 映射系统（2026-06-07 定稿）

**核心发现**：不是每个角色都需要单独跑 SPUM CLI 产线生成 prefab。代码中新增 `spum_template` 字段，JSON 填模板名即可映射到已有 SPUM prefab：

```json
{
  "spum_template": "hero_sword",   // 用剑客模板（西门吹雪外形）
  // 或 "hero_fist" — 拳掌模板（萧峰外形）
  // 或 "hero_archer" — 远程模板
  "effects": { "theme_color": "#9B59B6" }  // 门派色覆盖
}
```

**角色 JSON 中 art.spum_template 字段优先级**：
- 有 `spum_template` → 用模板 prefab，忽略 spum_parts
- 无 `spum_template` 但有 `spum_parts` → 走 CLI 产线
- 都无 → 强盗木桩兜底（不白方块）

**已注册模板**：
| 模板名 | 对应外形 | 适合 |
|--------|---------|------|
| hero_sword | 西门吹雪外形 | 剑客类：令狐冲、叶孤城、杨过 |
| hero_fist | 萧峰外形 | 拳掌类：虚竹、张无忌、郭靖 |

## VFX 特效架构（2026-06-08 定稿）

### 设计原则

1. **特效跟武功走，不跟人走** — VFX ID 存武功表或特效表，战斗引擎按武功名查表播放
2. **架构可替换，先程序后素材** — 第一版用 Unity ParticleSystem 程序生成，第二版同接口换成 Sprite 序列帧/VFX Graph
3. **8-10 原型 + 颜色/强度覆盖** — 定义「刀劲/拳气/暗器/内功/火/冰/雷/治愈/护盾/混沌」原型，每个武功JSON一行配置

### 接口分层

```
战斗引擎触发 VFX
  ↓ 通过 VFXID（如 "slash_sword" / "blast_fire" / "heal"）
  ↓
VFXResolver 查注册表
  ↓
    第一版：程序粒子实现（ParticleSystem，零资产依赖）
    第二版：同接口换成 Sprite 序列帧或 VFX Graph 实现（不调上游代码）
```

### VFX 方向：武器/流派差异化

| 武器类型 | VFX 语言 |
|---------|---------|
| 刀剑 | 斩击光刃 |
| 拳掌 | 气流冲击 |
| 暗器 | 飞针光点 |
| 法术/内功 | 元素粒子 |
| 大招 | 镜头拉近 + 顿帧 + 全屏波纹 |

### 不做：按角色逐个精调VFX（67角色×独立参数=工作量爆炸，留给外包阶段）

### ⚠️ SPUM AnimatorController 不可置空（2026-06-07 成哥实战经验）

**症状**：用 Animancer Pro 或其他代码动画系统控制 SPUM 角色时，置空 `runtimeAnimatorController` 导致角色卡死静止、部件不显示。

**根因**：SPUM 的 AnimatorController 不只是动画播放器——它还管理武器挂点激活、身体部件遮罩、材质参数同步、渲染组件激活时序。置空 = 丢了以上全部管线。

**修复**：不要动 `runtimeAnimatorController`，通过 `TryFindStateAndIndex + PlayAnimation(state, index)` 走 SPUM 原生状态机驱动。自定义动画用 OTHER 槽兜底。

详见 `references/SPUM动画集成经验.md`。

### ⚠️ SPUM 造形流程认知修正历史**（避免未来反复）：
1. 最初认为：art.spum 是文字描述，可自动喂给 SPUM 生成 → ❌ 错（SPUM 不是文字→外观引擎）
2. 被纠正为：SPUM 是 GUI 操作，LLM 做不了 → 这个认知被设为记忆
3. 后来用户文档（角色生产流水线.md）证实：**SPUM 外观可序列化，能用 CLI 脚本(SPUMCharacterBuilder)代码生成**——填 spum_parts(部件号+色码)即可，不依赖 GUI
4. **当前事实**：SPUM 造形已代码化，加/改武将=改 JSON + 跑一条命令。但 AI 看不到部件长相，需要人工 Play 校对→反馈调参。

**一律不用 Computer Use 操控 Unity**——截图点击循环烧 token 且不稳定。

### 伤害计算管道（2026-06-07 定稿 — 含触发率机制）

**公式来源**：玩家社区对原版武侠Q传的拆包分析，经108角色数据验证。
详见 `references/原版攻防公式.md` 和 `references/角色属性Build系统-定稿.md`。

```python
# -------- 属性计算 --------
最终atk = base_atk + growth_atk × (Lv-1) + 自由点武力 × 3
最终mp  = base_mp  + growth_mp  × (Lv-1) + 自由点内力 × 3
最终spd = base_spd + growth_spd × (Lv-1) + 自由点身法 × 2
最终def = base_def + growth_def × (Lv-1) + 自由点防御 × 5
最终hp  = base_hp  + growth_hp  × (Lv-1) + 自由点气血 × 10

# -------- 每回合触发判定（含身法加成） --------
真气 += 10(普攻) + 5(受击)
身法加成率 = 最终spd / 1000 × 身法系数    # 身法系数默认=1.0
实际触发率 = min(面板施展几率 + 身法加成率, 0.95)  # 上限95%
if 真气 ≥ 技能消耗 AND rand < 实际触发率:
    触发技能（走技能分支），真气 -= 技能消耗
else:
    普攻（走普攻分支）

# -------- 普攻分支 --------
普攻伤害 = 最终atk² / (最终atk + 最终def × 3)

# -------- 技能分支 --------
技能基础 = (最终atk × wa + 最终mp × wm + 最终spd × ws) × (range_multiplier / 100)
技能原伤 = 技能基础² / (技能基础 + 最终def × 3)

# AOE 位置衰减（当技能目标数 ≥ 2 时，逐个目标独立计算）
# 衰减系数: 第1目标100%、第2目标80%、第3目标60%
# 每个目标分别用自己的 def 和 mp 独立计算减伤和化解

# 化解判定（仅大招生效，普攻不吃化解）
if rand < 防守mp / (防守mp + 攻击mp) × 0.8:
    化解值 = max(防守mp - 攻击mp, 0) × 0.2  # 可选+面板化解
    技能终伤 = 技能原伤(衰减后) - 化解值

# -------- 统一收尾 --------
if rand < 0.15: 伤害 ×= 1.5        # 暴击
伤害 ×= (1 + 缘分%)                 # 缘分加成
伤害 ×= (1 + 格子属性%)             # 阵位加成
伤害 ×= rand(0.95, 1.05)            # 波动±5%
return max(1, round(伤害))

# -------- 回合顺序 --------
按最终spd从高到低排序
```

**关键参数：** C=3 | 化解基础率=0.8 | 内力化解系数=0.2 | 暴击率=15%(固定) | 波动=±5%

详见 `references/触发率速查.md`（基于成哥提供的85人原版技能数据重写）。每个武功有独立的 **面板施展几率**（如独孤九剑22%、罗汉拳35%），实际触发率 = 面板施展几率 + 最终身法/1000 × 身法系数。高身法角色（令狐冲60级spd=294，加成29.4%）的实战触发率远高于面板值。**⛔ 旧版文档中所有AI推断的触发率值（外功55-65%、内功35-50%等）已全部废除，不可引用。** 心法为被动无触发率。

**验证数据（60级，令狐冲全武力 vs 萧峰全防御）：**
- 普攻：216/击，活4.3击 ✅
- 技能（独孤九剑**面板22%→实战51.4%**，spd=294）：活约2.5击 ✅
  *注：旧版验证数据使用的55%为AI推断值，现已修正为面板22%+身法补正*

**伤害链逻辑：**
- 数值单一来源：JSON，无 ScriptableObject 中间层
- Prefab 不手编：全部由 CLI 从 JSON 生成
- SPUM_Adapter：反射驱动动画，不硬依赖 SPUM 类型
- 自由点存玩家存档，不存角色定义JSON

### V1（当前使用，待迁移）

| 字段 | 规范 |

### ⚠️ range字段误解——「贯穿63」不是破防值也不是威力指数（2026-06-07勘误）

**症状**：AI将武功总表 range 字段的数字（如「贯穿63」的63、「横扫18」的18）误用为「威力指数」来推导技能触发率，导致段誉六脉被推算为22%触发率（后修正为42%）。

**正确理解**：range 字段格式为 `<范围前缀><百分比系数>`：
- **前缀**：单体/贯穿(一列)/横扫(一排)/溅射/全体 → 决定AOE模式
- **数字**：百分比系数 → 技能基础威力 = 属性加权和 × (数字/100)
- 数字越大技能基础越强，与触发率无关

**影响范围**：
1. ❌ 之前写的 trigger_rate 推导公式（range≥60减8%等）**完全错误**，已移除
2. ❌ 伤害链公式原来漏了 `× (range_multiplier/100)`，已补上
3. ✅ 现在 trigger_rate 仅由 type(外功/内功/轻功) + quality(甲/乙) + 原版施展难度标注推断

**教训**：接手他人整理的数据时，不了解字段含义就自己编公式推导，是最大的风险。**应先向用户确认字段含义，或找原版文档验证，再动手。** 对于游戏中常见的 range/Multiplier/Power 等命名，不同游戏含义可能天差地别：
- 可能是百分比系数（本例）
- 可能是破防值
- 可能是技能等级上限
- 可能是固定数值加伤
- 不确定时不要猜，问。

详见 `references/武功总表range字段说明.md`。

### ⚠️ 触发率遗漏——「真气满必放」幻觉（2026-06-07修正）

**症状**：设计真气系统时默认为「真气满 → 自动放技能」，没考虑原版是概率触发。后果是内功角色（段誉六脉神剑）伤害爆炸且稳定触发，严重破坏平衡。

**根因**：
- 参考了梦幻西游「有蓝必放」的机制，但梦幻的平衡靠门派少+技能数值克制，不是靠概率
- 原版武侠Q传的技能触发率藏在角色天赋武功描述里（「较易施展」「不易施展」），我们之前没解析
- 真气系统的设计文档只写了「真气满→放技能」，完全没有提及触发概率

**修复**：（2026-06-07补入触发率机制，2026-06-08基于41条原版数据修正）\n- 真气系统保留作为释放门槛\n- 实际触发率 = **面板施展几率 + 最终身法 / 1000 × 身法系数(1.0)**\n- 面板施展几率从武功配置表读取（如独孤九剑22%、罗汉拳35%），每技能独立

**衍生问题：短战斗 × 低触发率 = 坐牢**（2026-06-07发现）\n修复过程中发现一个公式推导陷阱：段誉按「内功甲+贯穿63」算出的22%触发率，会导致5回合战斗有29%概率全程哑火。\n- 3v3战斗仅5回合，触发率 < 35% 时零触发概率超过10%，玩家体感明显\n- **核心设计铁律：短战斗(≤5回合)游戏中，核心角色触发率不应低于35%**\n- **触发率公式（基于range数值）仅供参考，必须用原版实际表现校准**。段誉在原版不弱，修正为42%后零触发概率仅6.6%\n- **教训**：任何「公式推导」的数值都必须在游戏实际长度和节奏下做期望验证。一个数学上合理的值可能在玩家体验中完全不合理。\n- **同样适用于range_multiplier**：内功角色的技能base_mul不能只按公式推，必须经实战验证输出量级是否匹配其定位

**教训**：设计回合制战斗系统时，**先确认原版机制再决定改与不改**。参考其他游戏的机制（如梦幻真气系统）时，要明确「这是全新设计还是对原版的改造」——如果是改造，必须说清楚改了哪里、为什么改、改后的平衡影响。触发率值不要迷信公式推导，用原版实际表现校准。

### ⚠️ 触发率与range数字解耦

### ⚠️ 新写文档的触发率系数陷阱（2026-06-08 发现）

**症状**：2026-06-08 审核角色设计文档.md和角色设计对接规范.md时，两份独立文档都把触发率公式写成 `spd/1000×0.1`，但定稿是「身法系数=1.0」。

**影响**：令狐冲(spd=294) 面板22%触发率：

### ⚠️ Unity C# 常见编译/运行时陷阱（2026-06-08 代码审查汇总）

#### 1. WaitUntil 布尔表达式缺运算符

**症状**：`yield return new WaitUntil(() => cond1 cond2 cond3)` 编译通过但逻辑错——C# 把三个并列条件解析为 `cond1 && !!cond2 && cond3`，实际永远是死循环。

**修复**：条件之间加 `||`：
```csharp
yield return new WaitUntil(() => cond1 || cond2 || cond3);
```

#### 2. 触发率公式遗漏身法加成

**症状**：`BattleManager.cs` 用 `ult.triggerChance` 直接 roll，忘了加 `+ spd/1000 × speedCoef`。段誉22%面板直接22% roll，实战变成22%而不是应有的55%。

**修复**：
```csharp
float chance = ult.triggerChance + (unit.SPD / 1000f) * CombatConfig.I.speedCoef;
if (chance < 0.05f) chance = 0.05f;
```

#### 3. Coroutine 定义了但从未启动

**症状**：`DrainUltimatesRoutine` 定义了完整的协程逻辑，但 `BattleLoop` 中从未 `yield return StartCoroutine(DrainUltimatesRoutine())`。玩家手动点大招按钮→压入队列→没人出队→永远不放。

**修复**：结算阶段结束后、`CheckEnd` 之前加启动调用。

#### 4. 自动站位写回遗漏

**症状**：`BattleBootstrap.SpawnEnemies` 中 auto-assign position 时把 `e.position` 改了但没写回 `enemies[i]`。后续敌人查 `enemies.Exists(x => x.position == ...)` 时仍读到旧值，导致冲突。

**修复**：
```csharp
enemies[i] = new EnemyEntry(e.characterId, e.level, Formation.EnemyFillOrder[autoIdx]);
```

#### 5. [Serializable] 遗漏（Unity 特有）

**症状**：`JsonUtility.FromJson<T>` 要求 T 标记 `[System.Serializable]`。遗漏时 Unity 不报运行时错误，但字段保持默认值（不填充），"已加载"日志也照打——静默失败。

**修复**：
```csharp
[System.Serializable]
public class CombatConfig { ... }
```

#### 6. damageMultiplier 单位确认

在 `DamageCalculator.cs` 中，`baseP = skill.damageMultiplier * (atk·aw + mp·mw + spd·sw)`。如果 JSON 中存的是整数百分比（六脉=63，西门一笑=413），当前代码会产生天文伤害。需确认单位是**小数**（0.63/4.13）还是**整数**（63/413），后者需补 `/= 100f`。

### ⚠️ CS1061 missing-field 模式（2026-06-07）

**症状**：`BattleManager.cs` 引用 `skill.targetMode`，但 `SkillDef` 没有定义该字段 → CS1061 编译错误。

**教训**：规则层新增类型/字段后，代码层必须先定义再使用。这是最常见的「文档先于代码」冲突——设计文档改了，但 C# 类定义没同步。修法：先在 SkillDef 补 `public TargetMode targetMode;`，再调编译器。

```
梦幻西游：
  12门派 × 每门派固定属性偏向 = 12种角色模板
  → 门派少 → 要靠堆叠系统来制造多样性
  → 所以有：修炼/经脉/装备/宝石/宠物/坐骑/法宝...

武侠Q传：
  64角色 × 每个有独特atk:mp:spd:def配比 = 64种天生不同的角色
  → 人物多 → 排列组合本身就够玩了
  → 不需要照搬梦幻那一大堆系统
```

**关键决策：**
- 64个角色的排列组合（6人队×9宫格×自由加点）才是核心护城河
- 加太多系统反而稀释"人物多"这个优势——玩家忙着点修炼，就没空研究阵容了
- 梦幻的"杂"是因为门派少，我们的"简"是因为人物多

## Data Handling Protocol（处理未知数据的铁律）

> 2026-06-07 定稿。原因：三次被纠正数据字段含义，根源都是同一类错误。

### 核心原则

**遇到不确定含义的数据字段时，先问用户，不自己编规则推导。**

### 具体步骤

1. **识别**：看到数值类字段（range/数字/系数）含义不确定时，标记 `# unknown`
2. **停**：不要基于这个字段做任何推导（触发率计算公式、平衡调整等）
3. **问**：问用户「这个字段的具体含义是什么？」
4. **验证**：拿到答案后，在测试脚本里跑一遍，确认结果合理
5. **固化**：将字段含义写入 reference 文档 + memory

### 本项目的已知字段约定

| 字段 | 含义 | 确认来源 |
|------|------|---------|
| range="贯穿63" | 范围前缀(贯穿=一列) + 百分比系数(63=63%) | 用户 |
| 面板施展几率 | 武功自带的固定施展概率（如独孤九剑22%、罗汉拳35%） | 用户提供41条Excel数据 |
| attr_add_rate | Lua数组`{atk_ratio, mp_ratio, spd_ratio}`，如独孤九剑`{0.4, 0, 0.6}` | 用户 |
| 身法系数 | 实际触发率公式中的spd加成系数，默认=1.0 | 用户（待确认） |
| talent_wugong | 角色天赋武功名（不一定是武功总表中的条目） | 数据本身 |
| growth=0 | 无升级数据（标记缺失，非设计意图） | 用户 |

### 常见陷阱

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| range 字段含数字 | 当威力指数推导触发率 | 只用来算技能基础威力 |
| 原版说「贯穿」 | 望文生义认为是穿排 | 查实测数据 |
| 做了触发率公式 | 用 range 数字修正触发率 | 触发率只由 type+quality+施展难度决定 |
| 战斗短(≤5回合) | 接受<35%触发率是合理设计 | 验证零触发概率，≤10%才可接受 |

## 两阶段指令制战斗系统（2026-06-07 定稿）

> 状态：设计稿完成，待开发
> 驱动问题：段誉类法系角色在实时制下普攻刮痧、大招看脸，缺乏每回合参与感

### 核心改动

从「实时按速度行动」改为梦幻西游式两阶段：

```
① 指令阶段：遍历己方存活单位，弹指令面板(攻击/武功/防御)，玩家选指令+目标
② 结算阶段：所有单位按SPD高->低，依次执行各自指令
```

速度只决定结算顺序，每单位每回合一次行动。

### 资源模型（方案2 — 折中）

| 资源类型 | 小武功 | 大招 |
|---------|--------|------|
| 消耗 | 冷却CD（统一2回合） | 攒怒气（保留现有系统） |
| 触发 | 指令阶段手动选 | 怒气满+rand < trigger_rate |
| 打断当前回合？ | 不打断 | 不打断，只标记就绪，下回合指令阶段让玩家选 |

### 武功槽扩展

```
CharacterDef {
  normalSkill: SkillDef        # 普攻（不变）
  ultimateSkill: SkillDef      # 大招（不变，攒怒+概率触发）
  wugong: List<SkillDef>       # 【新增】2-4个武功槽，每回合可选其一
  passive: PassiveDef?         # 【待填】专有概率被动
}
```

P0 每角色装1个武功，用武功总表已有数据：
- 段誉：装一阳指（小伤害，解决刮痧）
- 萧峰：装亢龙有悔（小范围攻击）
- 西门：装一剑西来（单体爆发）

### 效果类型扩展

| 类型 | 用途 | P0？ |
|------|------|------|
| Damage | 伤害（已有） | P0 |
| Heal | 回血（施法者mp越高回越多） | P0 |
| Shield | 护盾/保护 | P2 |
| Control | 封系（定身/睡/混乱） | P2 |
| Buff/Debuff | 临时改属性N回合 | P2 |

### Heal公式（新增定稿）

```
治疗量 = 施法者mp x Heal系数（0.3~0.5）
```

设计意图：内力高->回血多->站得住（段誉）。内力0->回血0->倒了就倒了（西门）。mp一个人扛了法攻、法防（化解）、奶量三个属性，属性价值拉满。

### 防御指令

```
本回合减伤~50% + 回怒+20（正常受击+5的4倍）
```

让防一手有战略价值，不做死按钮。

### 指令类型（MVP）

```
攻击（物理普攻·已有）
武功（从 wugong 选一个，按 effectType 执行）
防御（减伤+回怒）
（可选）逃跑
```

### 大招的触发率设计（重要规则）

```
怒气满时 -> 大招按钮高亮
玩家在指令阶段决定这回合放不放
选后走 trigger_rate 判定：
  触发 -> 播放技能动画
  未触发 -> 本回合改为普攻，怒气保持满
```

### 与梦幻西游的核心区别

| 维度 | 梦幻西游 | 武侠Q传 |
|------|---------|---------|

| 封系 | 核心玩法 | P2扩展，不做硬编码 |
| 自动战斗 | ALT+Q挂机 | 同上指令面板可选AI托管 |

## 9宫格战斗规则（2026-06-07 定稿）

```
[普攻规则] 优先打同列最前排，前排不倒打不到后排
[技能规则] 单体/贯穿/溅射/横扫/全体 见武功总表
[范围含义] range字段格式 `<前缀><系数%>`，如"贯穿63"=一列×63%
          详见 references/武功总表range字段说明.md
[前排价值] 强悍角色放前排=浪费输出潜力，但保护了后排
[贯穿含义] 贯穿/横扫打多个目标时各自独立计算减伤，位置衰减系数：第1个100%、第2个80%、第3个60%。非破防逻辑 —— 2026-06-07二次勘误
```

### 战斗节奏（真气+触发率系统）

```bash
开局真气：30
普攻+10，受击+5
技能消耗：小技能20-30 / 主技能40-60 / 大招70-100

每回合流程：
  ① 真气积累
  ② 判断 rand < trigger_rate?（真气足够时）
     → 触发 → 放技能，消耗真气
     → 不触发 → 普攻，真气保持

触发率影响节奏：
  实际触发率 = 面板施展几率 + 最终spd/1000 × 身法系数(1.0)
  面板施展几率分级参考（基于41条原版数据）：
    通用外功：22-35%（独孤九剑类低，罗汉拳35%最高）
    纯内力技：15-35%（谢逊狮子吼15%最低，天山折梅手35%最高）
    混合型：25-38%
    心法(被动)：100%，不参与判定
  实战中高身法角色（spd 250+）的实际触发率比面板高25%+

典型战斗进程：
  第1回合：普攻对轰 + 部分角色技能触发（运气好就放）
  第2回合：真气积累+概率触发混战
  第3-5回合：多轮触发对轰，哪边后排先倒哪边崩盘
  段誉在场时：可能一次六脉都没出就结束了 → 赌脸的真谛
```

**公式：**
```
普攻 = atk^2 / (atk + def x 3)
技能原伤 = base^2 / (base + def x 3)
技能终伤 = 技能原伤 - 化解值（化解判定：触发率=防守mp/(防守mp+攻击mp)x0.8，化解值=max(守mp-攻mp,0)x0.2）
```

- 64角色排列组合才是核心竞争力，不做深度系统堆叠
- 梦幻借鉴：只做武功修炼(P0)，不做天赋树/宠物/修炼线

### ⚠️ 天赋武功 vs 武功总表 — 两个独立系统（2026-06-08 重要教训）

**警告：绝对不要将天赋武功数据合并到武功总表中。它们是两个完全独立的系统。**

| 维度 | 天赋武功总表 | 武功总表 |
|------|------------|---------|
| 来源 | 角色出生自带的专属技能 | 打宝/掉落/学习 |
| 查询入口 | 按角色名查 | 按武功名查 |
| 触发率含义 | 面板%+spd补正=实战触发 | 固定值，无spd补正 |
| attr权重特点 | 极端（六脉0:10:0） | 温和（六脉0.4:0.6:0） |
| 更新方式 | 成哥补全 | 成哥网上查数据后更新 |

**同一个武功名在两个表里值不同**——六脉神剑在天赋武功里是段誉专属（22%面板+spd补正，attr 0:10:0），但在武功总表里是通用版（42%固定，attr 0.4:0.6:0）。合并会产生歧义。

**正确做法**：
- 天赋武功数据 → 写入 `references/天赋武功总表.json`
- 武功总表数据 → 写入 `references/武功总表.json`（与原版结构一致）
- 两个文件保持完全独立，不做交叉合并
- 如需关联查询，在文档中注明"该武功同时存在于武功总表"即可

## PvE 对手设计方法论（v2026-06-07）

> 详见 `references/PvE对手设计方法论.md`
> 验证脚本 `scripts/enemies_stage_1_10.py`
> 前10关定稿文档在 `/root/stock-data/wuxia-q/前10关对手设计_定稿.md`

### 核心原则

```
Lv1-3 教学层 → Lv4-5 过渡层 → Lv6-7 挑战层 → Lv8-10 困难层
每3关一个难度台阶，中间穿插教学关（引入新战斗维度）
```

### 设计流程

1. 确定关卡教学目的（速度/防御/法术/人数/Boss）
2. 从 `enemy_templates` 选敌人组合，覆盖2-3个维度
3. 通过 `scale_enemy(template, target_lv)` 缩放到关卡等级
4. 运行 `scripts/enemies_stage_1_10.py` 验证三条铁律：
   - ① 单次伤害 ≤ 玩家核心HP的12-15%
   - ② 击杀最硬敌人需3-6次普攻
   - ③ 总HP增长略快于玩家伤害增长

### PvE特有注意点（踩坑记录）

- **化解系统导致敌人技能失效**：玩家甲级角色mp高（萧峰200+），低mp敌人的技能被化解到≈0。解决方案：敌人技能走普攻(atk权重高)或配高mp
- **敌人atk在低数值区间敏感**：atk²/(atk+def×3)在atk<120时变化剧烈，早期关卡微调影响大
- **敌人技能也需要触发率**：敌人不能用「真气满必放」——他们在PvE中同样每回合按 `rand < trigger_rate` 判定。**设计敌人时请同时配置trigger_rate字段**，否则敌人技能永远不会触发或100%触发，二者都会破坏平衡（推荐敌人技能统一40-60%触发率）
- **⚠️ 数值体系切换——双方必须同步（2026-06-07新发现）**：
  当从大数值体系切换到小数值体系时，**玩家和敌人必须同时换**。本项目的实际教训：
  - 玩家：原版小数值（萧峰hp399/atk200）
  - 敌人：仍用早期大数值（山贼hp800、寨主hp2600）
  → 敌我量级差3-6倍，玩家要打8-12刀才杀一个杂兵
  → **任何公式/数值修改后，第一件事就是去敌人配置里把数值也改了**，否则测试全是无效数据
  → 详见 `scripts/enemies_stage_1_10.py` 的敌人模板
- **range_multiplier必须经Play校准（2026-06-07新发现）**：
  武功总表的range_mul值（如六脉神剑的63%）不能只看公式推导就定稿。段誉的range_mul=63%在游戏中导致他输出只有萧峰普攻的1/4，明显不合理。**任何公式推导的range_mul值都必须经实战验证**，尤其是内功角色（他们靠技能吃饭，技能系数不能给低）。发现不合理时直接在JSON里改，不要迷信推导值。
- **不做特殊机制**：中毒/封印/治疗等属于P2，前10关纯输出对拼
