# 游戏系统设计协作方法论

> 从武侠Q传属性Build系统设计过程中沉淀的工作流
> 适用范围：属性模型 / 战斗公式 / 升级养成 / 经济系统等游戏系统设计

---

## 一、用户偏好

成哥（项目负责人）在系统设计讨论中的偏好：

| 偏好 | 说明 |
|------|------|
| **数据驱动** | 给出具体数字、对比表、极端情况校验，不要空谈概念 |
| **对标分析** | 引用经典游戏（梦幻西游/原版武侠Q传）的做法做参照 |
| **定量比较** | 用表格对比不同方案的数值效果（如1点/级 vs 2点/级 vs 3点/级） |
| **决策记录** | 讨论出结果后立即写定稿文档，标记为「定稿✓」 |
| **战略先行** | 先讲清楚 "我们为什么要这样做" 的战略原则，再讲具体怎么做 |
| **不做清单** | 明确写出 "不做" 什么，避免未来反复讨论 |

## 二、标准工作流

```
Step 1: 理解需求
  └── 问清楚目标效果、参考游戏、现有数据

Step 2: 数据采集+分析
  └── 从原版数据反推公式（反推原始游戏的设计意图）
  └── 查找对标游戏的实现（梦幻西游等）
  └── 用Python/脚本做定量分析（攻防模拟、边缘测试）

Step 3: 提出方案
  └── 多个方案对比（表格展示优缺点）
  └── 给出推荐方案 + 理由

Step 4: 讨论确认
  └── 用户拍板或调整
  └── 确认关键决策点（数据库动不动的、数值大小的）

Step 5: 写入skill
  └── 更新定稿文档：references/角色属性Build系统-定稿.md
  └── 更新SKILL.md核心认知
  └── 更新memory
```

## 三、避坑指南

| 场景 | 该怎么做 |
|------|---------|
| 用户说"数据库定了不改" | ✅ 接受约束，在现有框架内设计方案 |
| 用户提到参考某游戏 | ✅ 先分析"它的核心逻辑是什么"，再想"我们能不能用" |
| 发现某系统不适用 | ✅ 明确说"不做"，并写理由 |
| 数值平衡参数 | ✅ 做极端情况校验（全点满 vs 不点），看会不会崩 |
| 用户给新URL/数据 | ✅ 能爬就爬，不能爬就问截图/结构 |

## 四、设计文档规范

每个系统定稿文档应包含：

1. **状态标记**：定稿✓ / 待讨论 / 待定
2. **顶层战略**：为什么这么做
3. **具体公式/模型**：完整无歧义
4. **数值校验**：极端情况分析
5. **不做清单**：明确排除的选项
6. **实施路径**：第一步做什么，第二步做什么
